## Problem

- The merged evolution slices can now plan runs, execute summary-only harness evaluations, create bounded isolated evaluation tasks, and materialize baseline/candidate inputs inside isolated task worktrees.
- The remaining gap is that the harness still does not execute real verify/test/metric commands inside those isolated worktrees. Metrics are still derived from dataset summaries or task-wrapper status rather than from a command receipt generated by the harness itself.
- Without a full worktree-backed harness executor, the evolution loop still cannot produce direct execution receipts, stdout/stderr captures, or real command-level runtime evidence for baseline/candidate comparisons.
- This slice must stay evaluation-only. It must not add production follow-up execution, promotion/invalidation recording, or CLI/MCP evolution ingress.

## Desired Outcome

- `src/sisyphus/evolution/harness.py` gains a full worktree-backed evaluation path on top of the existing bounded Sisyphus task/worktree setup.
- The executor derives a concrete command plan, runs those commands inside the isolated evaluation worktree, records structured execution receipts, and returns metrics/evidence based on actual command results.
- The result remains isolated to the evaluation worktree and does not mutate the live repository or widen authority into follow-up execution, approval, or promotion.
